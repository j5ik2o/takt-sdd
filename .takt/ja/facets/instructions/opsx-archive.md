完了したOpenSpec変更をアーカイブせよ。

**やること:**

1. タスクの説明から変更名を特定する
   - 名前が提供されている場合、そのまま使用する
   - 提供されていない場合、`openspec/changes/`でアクティブな変更を確認する（`archive/`ディレクトリを除外）
   - アクティブな変更が1つだけの場合、自動的に使用する
   - 複数存在する場合、リストアップしてABORTし、ユーザーに指定を求める

2. アーティファクトの完了ステータスを確認する
   ```bash
   openspec status --change "<name>" --json
   ```
   JSONを解析して以下を理解する:
   - `schemaName`: 使用されているワークフロー
   - `artifacts`: 各アーティファクトのステータスリスト

   `done`でないアーティファクトがある場合、警告として記録するが続行する。

3. タスクの完了ステータスを確認する
   変更ディレクトリ内のタスクファイル（通常`tasks.md`）を読む。
   `- [ ]`（未完了）と`- [x]`（完了）のタスクを数える。
   未完了タスクがある場合、警告として記録するが続行する。

4. デルタスペックの同期状態を評価する
   `openspec/changes/<name>/specs/`のデルタスペックを確認する。
   存在しない場合、ステップ5に進む。

   デルタスペックが存在する場合:
   - 各デルタスペックと`openspec/specs/<capability>/spec.md`の対応するメインスペックを比較する
   - 適用される変更（追加、変更、削除）を記録する
   - サマリーレポートに同期ステータスを含める

5. 公式のライフサイクルコマンドでアーカイブを実行する
   ```bash
   openspec archive "<name>" --yes
   ```
   このコマンドは変更の検証、デルタスペックの `openspec/specs/` への同期、
   `openspec/changes/archive/YYYY-MM-DD-<name>/` への移動をまとめて行う。

6. アーカイブの成功を確認する
   `openspec list` に対象変更が出なくなり、アーカイブ先ディレクトリが存在することを確認する。

**重要なルール:**

- アーカイブ前に必ずアーティファクトとタスクの完了ステータスを確認する
- 警告でアーカイブをブロックしない - レポートに含める
- sync と archive の挙動は `openspec archive --yes` を正とする
- アーカイブ後も .openspec.yaml が変更ディレクトリ内に保持されることを前提にする
